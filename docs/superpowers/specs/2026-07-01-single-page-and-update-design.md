# 单页抓取 + 已跟踪公司更新 —— 设计

日期：2026-07-01

## 背景

`campus-job-retrieval.md` 只覆盖"首次给一家公司建库"的全量列表页抓取场景。实际使用中还有两类没有固化流程的需求：

1. 用户直接粘一个具体职位详情页链接（非列表页），要求单条入库。
2. 用户点名一家已经跟踪过的公司，要求刷新——找出新增职位，同时知道哪些旧职位已经下架。

这次的目标是把这两类场景固化成 searcher 私有工作区下的 workflow 文档，并设计支撑它们所需的脚本改动。不涉及自动定时触发——本轮只做手动触发版本，定时化留作后续扩展。

## 目录结构

```
~/.claude/agents/searcher/workflows/campus-job/
├── campus-job-retrieval.md      # 已有：新公司/新方向首次建库，全量抓取
├── single-job-page.md           # 新增：粘一个具体职位链接，单条入库
└── update-tracked-company.md    # 新增：给已跟踪的公司重新抓一遍，找新增+标下架
```

三份文档共享 `~/Projects/campus-job-db/scripts/` 下的同一套脚本，只是编排方式不同——不重复实现抓取/结构化/写入逻辑。

## Notion Schema 变更

「2026校招实习职位汇总」数据源新增一个 select 字段 **状态**，选项 `在招` / `已下架`，默认 `在招`。

存量数据不强制批量回填：脚本对"没有这个字段值"的记录视为 `在招`，不需要一次性迁移写入。这个字段是一次性手动加（走 Notion UI 或 `notion-update-data-source` MCP 工具），不需要脚本自动建字段。

## `single-job-page.md`：单页抓取工作流

**触发**：用户粘一个具体职位详情页链接（不是列表页）。

**步骤**：
1. 抓取渲染后正文：API 优先；SPA 页面（如 mokahr/北森这类 hash 路由）退到 Playwright 渲染后取文本，避开直接 `WebFetch` 对纯前端渲染页面失效的问题。
2. 喂给 `split_jobs_from_raw.py`——即使只有一条职位也能跑，脚本靠模型识别边界，天然兼容单职位输入。
3. 可选：过 `interpret_jobs.py`（职位解读+资格提示）、`recommend_priority.py`（打分），和列表页流程规则一致，按用户是否给了求职意向/背景决定要不要跑。
4. `build_notion_pages.py --company <公司名>` 组装成 pages JSON。**不使用 `--link-template` 反拼链接**——单页场景原始 URL 已知，直接把用户给的链接原样写入 `链接` 属性，避免 job_id 反拼出错。
5. `push_notion_pages.py` 写入。`状态` 属性不传，走 Notion 侧默认值（等价于 `在招`）。

单条也走完整脚本链，不再临时手调 MCP 工具直接写页面正文——避免字段映射/去重逻辑和列表页路径产生两套实现。

## `update-tracked-company.md`：已跟踪公司刷新工作流

**触发**：用户点名一家已跟踪公司要求刷新（"查一下 Kimi 有没有新职位"），不做全量扫描、不新增公司。

**步骤**：
1. 复用 `campus-job-retrieval.md` 的抓取方法论，重新拉一遍该公司列表页，结构化后得到"本次抓到的完整职位链接集合"。
2. `push_notion_pages.py` 照常跑——现有的按 `链接` 去重逻辑保证只新增没见过的职位，这一步逻辑不变。
3. **新增步骤**：用"本次抓到的链接集合"对该公司在 Notion 里的既有记录做差集判定（见下面 `mark_delisted.py`）：
   - Notion 里有、但这次没抓到 → 标记 `已下架`
   - Notion 里已标 `已下架`、但这次又出现了 → 改回 `在招`（应对职位重新开放的情况，避免永久误标）

## 脚本改动

### 新增 `mark_delisted.py`

- 输入：`--company <公司名>`、`--current-links-file <json，本次抓取到的链接数组>`、`--database-id`。
- 逻辑：按 `公司` 属性过滤查询 Notion 现有记录（需要 `公司` 和 `链接` 和 `状态` 三个属性），与传入的链接集合做差集，对状态需要翻转的记录调用 Notion REST API 的 `PATCH /pages/{page_id}` 更新 `状态` 属性。
- 输出：打印本次标记为已下架 / 恢复在招的条数和职位名，便于核对。

### 修改 `push_notion_pages.py`

- `get_existing_links` 目前只取 `链接` 属性建 set。改成同时取 `公司` 属性，返回 `{link: company}` 或等价结构，供 `mark_delisted.py` 复用同一次查询结果，不用重复查一遍 Notion（避免两个脚本各查一次数据库）。
- 具体是把这个查询逻辑提出来做成两个脚本都能 import 的共享函数，还是 `mark_delisted.py` 独立实现一份同样的查询——留到写实现计划时再定，这里只确定"字段要多取一个"这个接口约束。

## 不在本轮范围内

- 定时/自动触发更新（scheduled-tasks 机制）——用户明确说先做手动版，接口层面不因此受影响，之后加定时触发只是换一个调用入口，不改这里设计的核心逻辑。
- 存量数据批量回填 `状态` 字段——按需求不强制做。
- 多公司一键批量刷新（"把所有跟踪的公司都刷一遍"）——用户明确了范围是"指定哪家抓哪家"，这个留作后续可能的扩展，不在本轮设计内。
